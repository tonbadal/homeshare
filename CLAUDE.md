# CLAUDE.md

## Project Overview
Shared family home management app. Families that co-own a property use this to coordinate bookings, share house instructions, manage tasks, and communicate.

## Tech Stack
- **Frontend:** Next.js 14+ (App Router), TypeScript, Tailwind CSS, shadcn/ui, lucide-react
- **Backend:** Supabase (Postgres, Auth, Edge Functions, Realtime, Storage)
- **Hosting:** Vercel
- **No ORM.** Use `@supabase/supabase-js` client directly.

## Project Structure
```
/app/(auth)/         → login, signup, invite redemption (public)
/app/(dashboard)/    → all authenticated routes
/lib/supabase/       → client.ts (browser), server.ts (SSR), middleware.ts
/lib/hooks/          → custom React hooks (useBookings, useTasks, etc.)
/lib/types/          → generated Supabase types (database.types.ts)
/components/ui/      → shadcn/ui components
/components/         → app-specific components
/supabase/migrations/→ SQL migration files (ordered by timestamp)
/supabase/functions/ → Edge Functions
/supabase/seed.sql   → development seed data
```

## Code Conventions
- Server components by default. Only use `"use client"` when the component needs interactivity (state, effects, event handlers, browser APIs).
- Use Supabase server client in server components and route handlers. Use browser client in client components.
- All dates stored as UTC in the database. Display in user's local timezone on the frontend.
- Use `date-fns` for date manipulation, never raw Date methods.
- Use shadcn/ui components as the base. Don't reinvent buttons, dialogs, dropdowns, etc.
- Keep components small and composable. One file = one component.
- Name files in kebab-case: `booking-calendar.tsx`, `task-list.tsx`.
- Name components in PascalCase: `BookingCalendar`, `TaskList`.
- Use TypeScript strictly. No `any` types. Generate DB types from the Supabase schema.

## Database Rules
- NEVER run production or remote migrations or database, only touch local supabase
- Every table must have RLS enabled.
- Every RLS policy must be scoped through `home_members` — a user can only access data for homes they belong to.
- Use `auth.uid()` in RLS policies, never pass user ID from the client.
- Foreign keys should cascade on delete where appropriate (e.g., deleting a home removes all its bookings, tasks, etc.).
- Use `uuid` for all primary keys, generated by `gen_random_uuid()`.
- Include `created_at` and `updated_at` timestamps on all main tables.
- When proposing and making migration changes, don't reset the database completely, prefer just upgrading to latest migration.

## Auth Rules
- All routes under `/(dashboard)` require authentication.
- Auth middleware refreshes the session on every request.
- Magic link is the preferred login method (simpler for non-tech users), email/password as fallback.
- Invite flow: user visits `/invite/[code]` → signs up or logs in → auto-joins the home with the role specified in the invite.

## Business Logic Placement
- Simple CRUD → Supabase client directly from Next.js (protected by RLS)
- Booking approval flow → Supabase Edge Function (validates policy, updates status, creates notifications)
- Invite redemption → Supabase Edge Function (validates code, creates membership, marks invite used)
- Notification fanout → Database trigger or Edge Function
- File uploads → Supabase Storage with bucket policies tied to home membership

## UI/UX Guidelines
- Mobile-first responsive design.
- Warm, friendly tone — this is a family app, not enterprise software.
- Use colour-coding for booking owners on the calendar (assign each member a colour).
- Pending bookings should look visually distinct from approved ones (faded/striped vs solid).
- Empty states should be helpful and encouraging, not blank.
- Loading states on all async operations (skeletons, spinners).
- Toast notifications for user actions (booking submitted, task completed, etc.).

## Environment Variables Needed

```
NEXT_PUBLIC_SUPABASE_URL=

SUPABASE_SERVICE_ROLE_KEY=
```

## Commands
- `npm run dev` — start Next.js dev server
- `npx supabase start` — start local Supabase (Docker required)
- `npx supabase db reset` — reset local DB and re-run migrations + seed
- `npx supabase gen types typescript --local > lib/types/database.types.ts` — regenerate types
- `npx supabase functions serve` — run Edge Functions locally
